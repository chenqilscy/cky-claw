# Kasaya 工具路线决策文档

> 版本：v1.0
> 日期：2026-04-16
> 状态：✅ 产品决策已达成
> 关联 Issue：#12 工具路线决策 · #21 工具组整体设计

## 1. 背景

Kasaya 平台存在四种工具来源，用户在配置 Agent 工具时面临选择困惑：

| 来源 | 入口 | 管理方式 |
|------|------|---------|
| **Hosted Tool**（内置工具） | 工具组 → 内置 | 系统自动种子，6 组标准工具 |
| **Custom Tool**（自定义工具） | 工具组 → 自定义 | ToolGroupPage 结构化编辑器 |
| **MCP Tool**（MCP 协议工具） | Agent 级 `mcp_servers` | MCP Server 管理页绑定 |
| **Agent-as-Tool** | Agent 级 `agent_tools` | Agent 编辑页第 4 步选择 |

#12 提出"弱化 MCP，最终不再支持 MCP Tool"；#21 要求"工具组整体设计重新规划"。

## 2. 决策结论

### 2.1 MCP 路线（#12）

**决策：保留 MCP，UI 层面降低展示优先级。**

| 维度 | 结论 |
|------|------|
| Framework 层 | ✅ 保持完整 MCP 支持（stdio/sse/http 三种传输） |
| Backend API | ✅ 保持 MCP Server CRUD + 连接测试 + 工具预览 |
| Agent 编辑 | ✅ MCP 配置在"编排配置"步骤（第 4 步），而非"工具配置"步骤（第 3 步） |
| Runner 合并 | ✅ 三路合并架构不变（Tool Groups + MCP + Agent-as-Tool） |

**理由：**

1. **行业标准**：MCP 是 Anthropic 推出的开放协议，OpenAI、Google 等主流厂商已支持。移除会丧失生态兼容性。
2. **向后兼容**：已发布的 API 和配置格式是承诺（核心信念"绝不破坏兼容"），移除 MCP 支持会导致现有 Agent 配置失效。
3. **用户选择**：对于已有 MCP Server 基础设施的企业用户，MCP 是低成本接入工具的方式。
4. **UI 优先级已实现**：Agent 创建步骤拆分后（#23），工具组在第 3 步"工具配置"，MCP 在第 4 步"编排配置"，天然形成主次关系。

## 3. 工具体系统一策略

### 3.1 用户使用推荐路径

```
首选 ─→ 工具组（Hosted + Custom）
         ├── 内置工具组（web-search, code-executor 等）开箱即用
         └── 自定义工具组（结构化编辑器或 JSON 定义）

次选 ─→ Agent-as-Tool
         └── 将专职 Agent 包装为工具，实现分工协作

高级 ─→ MCP Server
         └── 对接外部 MCP 兼容的工具服务（已有 MCP 基础设施的场景）
```

### 3.2 前端引导策略

| 位置 | 策略 |
|------|------|
| Agent 创建向导 | 第 3 步"工具配置"展示工具组（内置 + 自定义），带模板快速创建 |
| Agent 创建向导 | 第 4 步"编排配置"展示 MCP + Handoff + Agent-as-Tool（高级用户） |
| 工具组管理页 | 提供 5 个预置模板（网络搜索/代码执行/文件操作/HTTP/邮件） |
| 文档 | 推荐用户优先使用工具组，MCP 作为外部集成方案 |

## 4. 未来演进

| 阶段 | 方向 | 说明 |
|------|------|------|
| 当前 | 工具组 + MCP + Agent-as-Tool 三路并存 | 已实现 |
| 中期 | Skill Factory 与工具组融合 | Agent 自主创建的 Skill 可导出为工具组 |
| 远期 | MCP 作为工具组的数据源之一 | 将 MCP Server 的工具导入为 Custom 工具组 |
