  Agent Harness 架构研究报告

  1. 什么是 Agent Harness？

  Harness（线束/脚手架） 是介于 LLM（智能体）和可靠系统之间的 基础设施控制层。它不是 Agent 本身，而是围绕 Agent
  构建的管理、治理、状态持久化系统。

  ▎ 核心论点：自主性不是模型属性，而是架构属性。 2025 年让 Agent 变得可见，2026 年让 Harness 成为必然。

  2. 为什么需要 Harness？——可靠性的数学问题

  单步 95% 可靠性听起来很好，但：

  - 20 步链式执行：0.95^20 ≈ 36% 端到端可靠性
  - 200 步循环：失败几乎是必然

  Harness 的存在就是为了打断这条链——插入检查点、验证门、恢复机制，将脆弱的长链变成有界可验证的短单元。

  4. 四大核心组件

  (1) Guardrails（护栏）— 预防性约束

  - 能力边界、数据边界、工具作用域
  - 成本和时间预算、安全约束、输出 Schema
  - 关键原则：护栏放在 Runtime 里，不在 Prompt 里

  (2) Checkpoints（检查点）— 验证门

  - 单元测试、策略测试、Schema 验证、静态分析
  - 每次 Agent 运行前后的状态快照
  - "停止信任感觉，开始信任证据"的地方

  (3) Handoffs（交接）— 跨会话持久化

  - 不是摘要，是 控制文档：
    - 当前状态、变更内容、已验证项
    - 失败原因、下一步行动、证据链接
  - 文件系统是最通用的交接介质

  (4) Human Injection Points（人工注入点）

  - 审批不可逆操作、解决歧义、接受风险
  - 不是每步都介入，而是放在 防止级联失败的关键节点

  5. 上下文工程（Cache-First Design）

  这是 2026 Harness 最核心的技术革新：

  ┌────────┬─────────────────────────────┬────────────────────┐
  │  层级  │            策略             │      触发条件      │
  ├────────┼─────────────────────────────┼────────────────────┤
  │ Tier 0 │ 结构化输出                  │ 默认               │
  ├────────┼─────────────────────────────┼────────────────────┤
  │ Tier 1 │ 大结果立即外部化到 Artifact │ 工具返回 >8k token │
  ├────────┼─────────────────────────────┼────────────────────┤
  │ Tier 2 │ 延迟淘汰旧输入/结果         │ 上下文 80-90%      │
  ├────────┼─────────────────────────────┼────────────────────┤
  │ Tier 3 │ 压缩/摘要                   │ 窗口仍然不够       │
  ├────────┼─────────────────────────────┼────────────────────┤
  │ Tier 4 │ 全新窗口重启                │ 外部状态充分时     │
  └────────┴─────────────────────────────┴────────────────────┘

  缓存纪律：
  - 不在会话中动态增删工具（破坏缓存）
  - 不在会话中切换模型
  - 不重写 System Prompt 反映动态状态
  - 保持序列化确定性

  6. 子智能体（Subagent）策略

  第一原则：不要一开始就用多 Agent。

  ┌───────────┬─────────────────────────────┬──────────────┐
  │   模式    │          适用场景           │     代价     │
  ├───────────┼─────────────────────────────┼──────────────┤
  │ 单 Agent  │ 大多数任务                  │ 最简单       │
  ├───────────┼─────────────────────────────┼──────────────┤
  │ Skills    │ 一个 Agent 需要大量隐式能力 │ 上下文累积   │
  ├───────────┼─────────────────────────────┼──────────────┤
  │ Subagents │ 需要上下文隔离/并行/专业化  │ 额外编排开销 │
  ├───────────┼─────────────────────────────┼──────────────┤
  │ Handoffs  │ 顺序阶段式对话              │ 更难推理     │
  └───────────┴─────────────────────────────┴──────────────┘

  默认模式：Orchestrator-Worker（编排者-工作者），非 Peer-to-Peer。

  9. Kasaya 可以借鉴的改进方向

  基于 Harness 蓝图，Kasaya 可考虑的增强：

  1. Artifact Store 外部化 — 大型工具输出写出到文件系统/对象存储，只返回引用
  2. Cache-First Prompt 布局 — 固定 System Prompt 前缀 + 追加式历史
  3. 结构化 Handoff 文档 — 跨会话交接的标准化格式
  4. 分层审批引擎 — Hook → 静态规则 → 权限模式 → 运行时回调
  5. Compaction + Fresh Restart 策略 — 上下文压缩 + 条件性全新窗口
  6. Instrumentation 完善 — 追踪缓存命中率、压缩频率、工具选择准确率

  10. 关键反模式（要避免的）

  1. 把 Agent 当作"只是一个 Prompt"
  2. 会话中动态改写工具集
  3. 让原始工具输出淹没上下文
  4. 因为"听起来高级"就用多 Agent
  5. 依赖模型自我约束风险操作
  6. 对模型隐藏错误信息
  7. 跳过对 Harness 本身的评估
